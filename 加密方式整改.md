# 加密方式整改说明

## 1. 加密解密方式统一整改

### 1.1 只保留一种加密方式
- 全部采用 AES-256-GCM 算法，IV 长度 12 字节，认证标签 16 字节，密钥长度 32 字节（支持 base64）。
- 加密输出格式为：`Base64(IV + EncryptedData + AuthTag)`，即所有加密数据都是标准 base64 字符串。

### 1.2 代码层面整改
- `src/lib/encryption.ts` 只保留一种加密/解密实现，所有加密、解密、重新加密、格式校验等逻辑全部集中于此文件。
- 删除了所有冗余、历史、兼容性加解密实现（如 `src/lib/decryptIdCard.ts` 等）。
- `encryptIdCard`、`decryptIdCard`、`reencryptData` 等函数全部只支持新格式，且有详细的错误处理和日志输出。
- 任何解密失败、密钥未初始化、格式不符等问题都会有详细日志，便于排查。

### 1.3 密钥管理
- 统一通过环境变量 `ENCRYPTION_KEY` 和 `OLD_ENCRYPTION_KEY` 管理密钥，支持 base64 和明文（自动转为 32 字节）。
- 所有环境（开发、测试、生产）都需保证密钥一致，否则无法解密历史数据。

### 1.4 数据迁移
- 编写了自动化迁移脚本 `src/scripts/migrate-encryption.ts`，对所有历史数据（客户、申诉、用户表）进行批量重新加密，统一为新格式。
- 迁移脚本会自动跳过已是新格式的数据，遇到异常会保留原数据并输出警告，保证数据安全。

---

## 2. 整改结果

- 所有历史数据已批量迁移为新加密格式，数据库中不再有混合格式。
- 所有加密/解密调用都指向同一份代码，无论是 API、后台、管理端还是服务端渲染。
- 密钥管理规范，只需维护 `.env` 文件中的密钥即可。
- 后台日志清晰，遇到任何解密失败、密钥问题、格式问题都能快速定位。

---

## 3. 注意事项

1. 密钥必须一致：所有环境的 ENCRYPTION_KEY 必须和迁移时用的一致，否则无法解密。
2. 新数据自动加密：所有新录入的身份证号都会自动用新方式加密。
3. 如需更换密钥：请先设置 OLD_ENCRYPTION_KEY，再批量迁移数据，最后切换 ENCRYPTION_KEY。
4. 如遇解密失败：请第一时间检查密钥和数据格式。

---

## 4. 结论

- 现在所有后台、API、管理端都只会用统一的加密解密方式，超级管理员和所有接口都能正常解密身份证号。
- 数据安全性和一致性大幅提升，维护和排查也更简单。
