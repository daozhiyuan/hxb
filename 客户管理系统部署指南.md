# 客户管理系统部署指南

本文档详细说明了如何部署和管理客户管理系统，特别针对证件号码加密解密功能的部署需求进行了优化。

## 一、系统要求

- Docker 20.10+
- Docker Compose 2.0+
- 至少2GB内存
- 20GB可用磁盘空间

## 二、Docker配置优化说明

系统的Docker部署配置已经过全面优化，主要改进包括：

1. **统一端口配置**：将Dockerfile和docker-compose.yml中的端口统一为3005
2. **加密环境变量**：添加ENCRYPTION_KEY和OLD_ENCRYPTION_KEY环境变量，支持证件号码加密解密
3. **持久化存储**：为上传文件和日志添加卷映射
4. **健康检查**：添加数据库和应用服务的健康检查
5. **数据备份**：增加自动备份服务，每日备份数据库并保留最近7天的备份
6. **安全性增强**：改进了入口脚本，增加了更多健壮性和安全性检查

### 主要优化文件

| 文件名 | 优化内容 |
|--------|---------|
| docker-compose.yml | 统一端口、加密环境变量、健康检查、备份服务、持久化卷 |
| Dockerfile | 添加工具包、目录权限、健康检查 |
| docker-entrypoint.sh | 增强启动脚本、健康检查API、环境变量检查 |
| deploy.sh | 一键部署脚本，自动化整个部署过程 |

## 三、一键部署流程

最简单的部署方式是使用一键部署脚本：

```bash
# 获取代码
git clone https://github.com/daozhiyuan/keyingbao.git
cd keyingbao

# 给部署脚本添加执行权限
chmod +x deploy.sh

# 运行一键部署脚本
./deploy.sh
```

一键部署脚本会自动完成以下操作：
- 检查Docker和Docker Compose是否安装
- 创建必要的目录结构（backups、logs、uploads等）
- 自动生成安全的随机密码和加密密钥
- 配置环境变量文件（.env）
- 更新Docker配置文件中的环境变量
- 配置数据库初始化脚本
- 启动服务并等待服务就绪
- 提供部署成功提示和后续操作指导

### 手动配置方式

如果需要手动配置，请按以下步骤操作：

#### 1. 获取代码

```bash
git clone https://github.com/daozhiyuan/keyingbao.git
cd keyingbao
```

#### 2. 配置环境变量

创建`.env`文件，添加以下内容：

```
# 数据库配置
MYSQL_ROOT_PASSWORD=设置强密码
MYSQL_DATABASE=nextn

# 加密密钥（必须配置，用于证件号码加密解密）
ENCRYPTION_KEY=生成32位随机字符串
OLD_ENCRYPTION_KEY=旧加密密钥（如有系统升级）

# NextAuth配置
NEXTAUTH_SECRET=设置一个随机字符串
NEXTAUTH_URL=http://你的域名:3005

# 系统配置
NODE_ENV=production
```

> **重要提示**：ENCRYPTION_KEY用于敏感数据加密（证件号码等），必须安全保存，一旦丢失将无法解密数据。

#### 3. 创建必要目录

```bash
mkdir -p ./backups ./logs ./public/uploads ./mysql-init
chmod -R 777 ./public/uploads ./logs ./backups
```

#### 4. 启动服务

```bash
# 使用Docker Compose启动所有服务
docker-compose up -d

# 仅启动MySQL（开发测试用）
docker-compose -f docker-compose-mysql.yml up -d
```

#### 5. 检查服务状态

```bash
docker-compose ps
```

服务正常启动后，可通过以下地址访问：

- 应用界面：http://服务器IP:3005

## 四、证件号码加密解密功能部署

系统的证件号码加密解密功能是核心安全特性，需要特别注意以下部署要点：

### 1. 加密密钥配置

- ENCRYPTION_KEY必须设置，且为32字节长度的强密钥
- 建议使用`openssl rand -base64 32`生成随机密钥
- 密钥必须妥善保存，丢失将导致无法解密已加密数据

### 2. 支持多种加密格式

当前系统支持以下加密格式：
- 标准格式：`IV:EncryptedHex`
- Base64格式：全部加密数据（含IV）的Base64编码
- URL安全Base64格式：替换了`+`和`/`的特殊Base64编码
- 特殊格式：以`tkl1`开头的加密数据

### 3. 迁移与兼容性

- 如更换加密密钥，请将旧密钥保存到OLD_ENCRYPTION_KEY环境变量
- 系统会自动尝试使用旧密钥解密，确保数据平滑迁移
- docker-entrypoint.sh脚本会检查加密密钥配置是否正确

### 4. 超级管理员解密工具

- 超级管理员可通过个人中心访问专用解密工具
- 此工具支持所有加密格式的解密，包括特殊格式
- 系统会自动识别加密格式并尝试相应的解密方法

## 五、数据备份与恢复

### 自动备份

系统已配置自动每日备份数据库，备份文件存储在`./backups`目录，保留最近7天的备份。备份服务在docker-compose.yml中配置：

```yaml
# Backup service
backup:
  image: mysql:8.0
  volumes:
    - mysql-data:/var/lib/mysql:ro
    - ./backups:/backups
  environment:
    - MYSQL_PWD=password
  command: >
    bash -c '
    while true; do
      BACKUP_FILE="/backups/nextn_$$(date +"%Y%m%d_%H%M%S").sql";
      mysqldump -h db -u root nextn > $$BACKUP_FILE;
      echo "数据库备份完成: $$BACKUP_FILE";
      find /backups -name "nextn_*.sql" -type f -mtime +7 -delete;
      echo "已删除7天前的旧备份";
      sleep 86400;
    done
    '
```

### 手动备份

```bash
# 手动备份数据库
docker-compose exec backup sh -c 'mysqldump -h db -u root -p$MYSQL_ROOT_PASSWORD nextn > /backups/manual_backup_$(date +"%Y%m%d_%H%M%S").sql'

# 备份上传的文件
tar -czf uploads_backup_$(date +"%Y%m%d").tar.gz ./public/uploads
```

### 数据恢复

```bash
# 从备份文件恢复数据库
docker-compose exec db sh -c 'mysql -u root -p$MYSQL_ROOT_PASSWORD nextn < /path/to/backup.sql'

# 恢复上传文件
tar -xzf uploads_backup_YYYYMMDD.tar.gz -C ./
```

## 七、常见问题排查

### 1. 系统无法启动

检查日志：
```bash
docker-compose logs app
```

常见原因及解决方案：
- 数据库连接失败：检查数据库服务是否正常运行
- 环境变量配置错误：检查`.env`文件配置
- 端口冲突：确认3005端口未被占用

### 2. 证件号码解密失败

可能原因：
- ENCRYPTION_KEY配置错误
- 数据可能使用了旧密钥加密
- 特殊格式加密数据需要使用专用解密工具

解决方案：
- 确保设置了正确的ENCRYPTION_KEY
- 对旧数据，配置OLD_ENCRYPTION_KEY
- 超级管理员使用专用解密工具（在个人中心）
- 检查数据是否为特殊格式（tkl1开头）

### 3. 文件上传失败

可能原因：
- 权限问题
- 存储空间不足

解决方案：
- 检查uploads目录权限：`chmod -R 777 ./public/uploads`
- 清理磁盘空间

## 11. 研发与整改变更梳理

### 1. 主要涉及的文件路径

| 文件路径 | 说明 |
|----------|------|
| `src/lib/encryption.ts` | 证件号码加密解密核心逻辑（已统一为RSA+AES混合加密，兼容多格式） |
| `src/scripts/generate-keys.ts` | 生成RSA密钥对脚本 |
| `src/scripts/migrate-to-rsa.ts` | 数据批量迁移为新加密格式脚本（自动检测字段） |
| `src/app/api/*/route.ts` | 相关API接口，涉及客户、申诉、注册、解密等 |
| `package.json` | 新增/调整脚本命令（generate:keys、migrate:rsa等） |
| `next.config.js` | Next.js服务端配置，支持ESM语法 |
| `docker-compose.yml`、`docker-compose-mysql.yml` | Docker服务编排与数据库配置 |
| `客户管理系统部署指南.md` | 本文档，部署与运维说明 |
| `keys/public.pem`、`keys/private.pem` | RSA公私钥文件（自动生成） |

### 2. 项目开发语言与技术栈

- **后端/服务端**：  
  - Node.js (TypeScript)
  - Next.js 14（API路由、SSR/SSG）
  - Prisma ORM（数据库操作）
  - MySQL 8.x
  - PM2（进程管理）
  - Docker & Docker Compose（容器化部署）

- **前端**：  
  - React 18
  - Next.js 14（同构渲染）
  - Tailwind CSS（UI样式）
  - Radix UI、Lucide React（组件库）
  - React Hook Form（表单校验）

- **安全与加密**：  
  - AES-256-GCM（对称加密，已升级为混合加密）
  - RSA-2048（非对称加密，密钥分发与数据保护）
  - OpenSSL（密钥生成）

### 3. 主要API与业务代码说明

- **加密/解密核心**：`src/lib/encryption.ts`
  - `encryptIdCard`：自动生成AES密钥加密证件号，再用RSA公钥加密AES密钥，输出JSON格式
  - `decryptIdCard`：自动识别JSON格式，先用RSA私钥解AES密钥，再用AES解密数据，兼容明文/老格式
  - `reencryptData`：批量迁移老数据为新格式
  - 兼容多证件类型（身份证、护照、港澳证、外国证件）

- **密钥管理与迁移脚本**：
  - `src/scripts/generate-keys.ts`：一键生成RSA密钥对
  - `src/scripts/migrate-to-rsa.ts`：自动检测表字段，批量迁移所有证件号为新加密格式

- **API接口**（部分示例）：
  - `src/app/api/customers/register/route.ts`：客户注册，证件号加密存储
  - `src/app/api/admin/customers/[id]/route.ts`：客户详情，证件号解密展示
  - `src/app/api/appeals/[id]/route.ts`：申诉详情，证件号解密展示
  - `src/app/api/admin/decrypt-idcard/route.ts`：超级管理员专用解密工具，支持所有格式

- **前端调用**：
  - 通过API获取数据后，前端只展示后端解密结果，不直接处理加密逻辑，保证安全性

### 4. 部署与运维要点（补充）

- **密钥管理**：`keys/`目录下的`public.pem`和`private.pem`为RSA密钥，务必妥善备份，防止丢失
- **数据库字段**：证件号相关字段（如`idCardNumberEncrypted`、`idNumber`）需为`TEXT`类型，避免密文过长导致存储失败
- **环境变量**：`.env`需配置数据库、加密密钥、NextAuth等，详见文档第3节
- **升级与迁移**：如有历史数据，务必先执行`npm run generate:keys`和`npm run migrate:rsa`，确保数据格式统一
- **日志与监控**：通过`pm2 logs`、`docker-compose logs`等命令实时监控服务状态和解密报错
