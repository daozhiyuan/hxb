# 域名登录跳转问题修复记录

## 1. 初始问题：登录后跳转到 localhost

*   **现象**：用户在生产环境 (bb.keti.eu.org) 登录或登出后，被重定向到 `http://localhost:3000` 而不是正确的域名。
*   **原因**：
    *   `next.config.js` 和 `ecosystem.config.cjs` (后被移除) 中的 `NEXTAUTH_URL` 和 `NEXT_PUBLIC_APP_URL` 配置不正确，或者没有正确传递到应用运行时。
*   **修复**：
    *   在 `next.config.js` 中确保 `basePath` 和 `assetPrefix` (如果使用) 指向正确的域名。
    *   确保 `NEXTAUTH_URL` 和 `NEXT_PUBLIC_APP_URL` 环境变量在 Docker 容器中被正确设置并传递给 Next.js 应用。我们在 `docker-compose.yml` 中设置了这些：
        ```yaml
        environment:
          - NODE_ENV=production
          - NEXTAUTH_URL=https://bb.keti.eu.org
          - NEXT_PUBLIC_APP_URL=https://bb.keti.eu.org
        ```
    *   更新 `src/middleware.ts`，确保在重定向到登录页时，`baseUrl` 使用的是 `process.env.NEXTAUTH_URL`。

## 2. 问题：登录成功后不跳转，或跳转回登录页 (ERR_INVALID_URL)

*   **现象**：即使用户凭证正确，登录后页面不发生跳转，或者跳转后又回到了登录页。浏览器或后端日志出现 `ERR_INVALID_URL`，输入为类似 `%2Fdashboard` 的编码后路径。
*   **原因**：
    1.  **双重编码 `callbackUrl`**：在 `src/middleware.ts` 中，当用户未登录需要重定向到登录页时，`callbackUrl` (即原始请求路径，如 `/dashboard`) 被 `encodeURIComponent` 编码了一次。之后，`next-auth` 内部或客户端 `signIn` 时可能又对这个已经编码的 `callbackUrl` 进行了处理，导致在尝试构建最终重定向URL时，URL变为无效格式 (如 `https://bb.keti.eu.org/%2Fdashboard` 被再次处理成类似 `https://bb.keti.eu.org/%252Fdashboard`，或者 `new URL('%2Fdashboard')` 直接失败)。
    2.  **`next-auth` 的 `redirect` 回调处理**：`next-auth` 的 `redirect` 回调接收到的 `url` 参数可能是相对路径 (如 `/dashboard`)，也可能是已经编码过的路径 (如 `%2Fdashboard`)。如果回调没有正确处理这些情况并构建一个绝对且有效的URL，会导致 `ERR_INVALID_URL`。
    3.  **`signIn` 函数的 `callbackUrl`**：在登录页面 (`src/app/login/page.tsx`) 调用 `signIn` 函数时，传递的 `callbackUrl` 如果是从查询参数中获取且未解码，也可能导致问题。
*   **修复**：
    *   **移除 `src/middleware.ts` 中的 `encodeURIComponent`**：
        ```typescript
        // src/middleware.ts
        // ...
        const fullPath = request.nextUrl.pathname + request.nextUrl.search;
        loginUrl.searchParams.set('callbackUrl', fullPath); // 移除了 encodeURIComponent
        // ...
        ```
    *   **增强 `src/lib/auth.ts` 中的 `redirect` 回调**：
        *   增加了对传入 `url` 的 `decodeURIComponent` 尝试，以处理像 `%2Fdashboard` 这样的输入。
        *   确保最终返回的是一个基于 `baseUrl` 的绝对且有效的 URL。
        ```typescript
        // src/lib/auth.ts (authOptions.callbacks.redirect)
        async redirect({ url, baseUrl }) {
          console.log('[Auth Redirect Callback] Original URL:', url);
          console.log('[Auth Redirect Callback] Base URL:', baseUrl);

          let decodedUrl = url;
          try {
            decodedUrl = decodeURIComponent(url);
          } catch (e) {
            console.warn('[Auth Redirect Callback] Failed to decode URL component:', url, e);
          }
          console.log('[Auth Redirect Callback] Decoded URL (if applicable):', decodedUrl);

          if (decodedUrl.startsWith('/')) {
            const finalUrl = `${baseUrl}${decodedUrl}`;
            console.log('[Auth Redirect Callback] Decoded URL is relative, returning:', finalUrl);
            return finalUrl;
          }
          
          try {
            const parsedUrl = new URL(url);
            if (parsedUrl.origin === new URL(baseUrl).origin) {
              console.log('[Auth Redirect Callback] Original URL is absolute and matches baseUrl origin, returning:', url);
              return url;
            }
          } catch (e) { /* ... */ }

          try {
            const parsedDecodedUrl = new URL(decodedUrl);
            if (parsedDecodedUrl.origin === new URL(baseUrl).origin) {
              console.log('[Auth Redirect Callback] Decoded URL is absolute and matches baseUrl origin, returning:', decodedUrl);
              return decodedUrl;
            }
          } catch (e) { /* ... */ }

          console.log('[Auth Redirect Callback] URL is external or unexpected. Defaulting to baseUrl:', baseUrl);
          return baseUrl;
        }
        ```
    *   **客户端 `signIn` 调用**：确保在 `src/app/login/page.tsx` 中，从 `searchParams.get('callbackUrl')` 获取的 `callbackUrl` 如果被编码过，`next-auth` 的 `signIn` 应该能够正确处理或者在传递前进行解码（虽然在这个案例中，后端的 `redirect` 回调处理更关键）。

## 3. 问题：登录成功，但中间件判定 Token 无效，导致重定向回登录页（循环）

*   **现象**：用户认证成功，JWT和会话都已生成。但当浏览器重定向到 `/dashboard` 后，`src/middleware.ts` 中的 `getToken()` 返回 `null`，导致中间件认为用户未登录，又重定向回登录页。
*   **原因**：`getToken()` 函数无法正确解析会话 cookie。这通常发生在使用自定义 cookie 配置（如 `secure: true`, `sameSite: "none"`, 或自定义 `cookieName`) 时，而 `getToken()` 调用时没有提供匹配的参数。
*   **修复**：
    *   在 `src/middleware.ts` 中调用 `getToken()` 时，明确指定 `cookieName`，使其与 `src/lib/auth.ts` 中 `authOptions.cookies.sessionToken.name` 的配置一致。
        ```typescript
        // src/middleware.ts
        // ...
        const cookieName = 'next-auth.session-token'; // 与 authOptions 中定义的 cookie 名称一致
        
        const token = await getToken({ 
          req: request, 
          secret: process.env.NEXTAUTH_SECRET,
          cookieName: cookieName, 
        });
        // ...
        ```
    *   确保 `authOptions` 中的 cookie 配置（特别是 `sessionToken`) 设置了 `secure: true` 和 `sameSite: "none"`，并且 `path: "/"`，这对于跨域HTTPS环境下的 cookie 至关重要。
        ```typescript
        // src/lib/auth.ts
        cookies: {
          sessionToken: {
            name: `next-auth.session-token`,
            options: {
              httpOnly: true,
              sameSite: "none", // Crucial for cross-site contexts
              path: "/",
              secure: true      // Crucial for SameSite=None
            },
          },
          // ... other cookies (callbackUrl, csrfToken) similarly configured
        }
        ```

## 4. 其他相关配置和排查点

*   **`SESSION_SECRET` 和 `COOKIE_SECRET`**：虽然在 `next-auth` v4 中主要是 `NEXTAUTH_SECRET`，但确保相关的会话和 cookie 密钥在环境变量中正确设置。
*   **Docker 容器的 `KeyError: 'ContainerConfig'**：在多次构建和重启过程中遇到此错误，通常通过 `docker-compose down --remove-orphans && docker-compose up -d --build` 彻底清理容器和网络后解决。
*   **浏览器缓存**：在调试此类问题时，务必清除浏览器缓存和 Cookie，或使用无痕模式进行测试，以避免旧的 Cookie 或缓存状态干扰。

通过以上步骤，域名登录和跳转问题最终得到解决。
